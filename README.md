## Philosophy

Open source is where I’ve learned the most about software.

You step into codebases written by people you’ve never met.  
You try to understand decisions made years ago.  
You follow the reasoning behind design choices, patterns, and trade-offs.

At first it feels unfamiliar.

But slowly you begin to see the system as a whole.

You start fixing small things.  
Then improving pieces of the architecture.  
Then contributing changes that make the project better for everyone who uses it.

Doing this repeatedly teaches something no tutorial ever can — how real software evolves over time.

That process is what shaped the way I think about engineering.

## Work in the Open

Much of my engineering work happens inside public repositories.

One project I have contributed to extensively is **Checkstyle** — a widely used static analysis tool that helps enforce code quality across Java projects used by developers worldwide.

My work appears across multiple releases of the project and focuses on improving the stability, maintainability, and developer experience of the codebase.

Contributions include work on:

• improving CI pipeline reliability and build stability  
• refining documentation and developer onboarding materials  
• strengthening test infrastructure and reducing suppressions  
• improving formatting consistency and codebase clarity  
• addressing cross-platform development issues  
• maintaining long-term repository health

Working inside mature projects like Checkstyle requires understanding large codebases, respecting long-standing architectural decisions, and contributing improvements that integrate cleanly with an evolving ecosystem.
